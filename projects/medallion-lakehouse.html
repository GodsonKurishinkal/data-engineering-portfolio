01

The problem

Supply chain operations ran on a patchwork of ERP, WMS, OBI, and a small army of Excel exports. Each source had its own shape, its own cadence, and its own definition of truth. The cost was obvious to anyone sitting at a desk.

  • Daily data extraction consumed 4+ hours before analysis could even start.
  • Reports were routinely 24–48 hours stale, blocking same-day decisions.
  • Analysts spent 60% of their time reconciling fields, not answering questions.
  • No lineage. Every incident became a forensic exercise.

The team didn't need a bigger dashboard. They needed a platform that gave them data they could trust, at a cadence that matched how the business actually ran.

02

The architecture

A medallion lakehouse — three layers, each with a clear contract. Bronze is the raw-source record of truth. Silver is clean, conformed, and validated. Gold is the business-ready star schema that powers everything from Power BI semantic models to forecasting notebooks.

Bronze — raw landing zone. Every source dumps in its native shape; nothing is transformed. This is the immutable historical record that lets us reprocess anything, anytime.

Silver — schema-enforced, deduplicated, typed. The 3-tier anomaly detection system lives here, catching 500+ dimension issues before they corrupt Gold.

Gold — Kimball star schema with 10+ fact tables and 6+ conformed dimensions. Optimised for analytical queries and semantic layers. One place for business logic; everything downstream inherits it.

03

Key decisions

Polars over PandasADR-001

Silver transforms on millions of rows per night needed to finish in a window, not a shift. Polars' Arrow-backed expression engine ran 5–10× faster than Pandas on the same workloads, with a lazier memory profile.

Trade-off: smaller community, fewer Stack Overflow answers. Worth it — the API is narrower and more predictable.
Hive-partitioned Parquet, not DeltaADR-002

We weren't running on Spark or Databricks. A plain Parquet lakehouse partitioned by extract_date gave us the query speed we needed via DuckDB + Polars with zero platform lock-in.

Trade-off: no time-travel or ACID guarantees on Bronze. Backfills run via explicit partition overwrite.
Configuration-driven ETLADR-003

Every pipeline is a YAML config + an Abstract Base Class. New sources go live in hours, not days. Schema, quality rules, and load mode (FULL / INCREMENTAL) are declarative.

Trade-off: steeper ramp for contributors. Mitigated with templates and reviewed PRs.
Windows Task Scheduler for orchestrationADR-004

Airflow would have been heavier than the platform itself. We used the OS scheduler already paid for, with structured logs and Teams alerts on failure.

Trade-off: no DAG visualization. Solved with a small Streamlit status board reading from the pipeline log table.
04

Implementation

Each pipeline inherits from a base extractor/transformer/loader trio. Behavior is declared in a YAML file; the engine reads it and runs the pipeline. That keeps the 50+ pipelines consistent and makes behavioral changes reviewable as text diffs, not code reviews.

pipelines/inventory_daily.yaml YAML 
# Declarative pipeline — the engine reads this, not a script.
pipeline:
  name: "inventory_daily"
  source:
    type: "oracle_obi"
    query: "SELECT * FROM vw_inventory_snapshot WHERE snapshot_date = :d"
  destination:
    layer: "bronze"
    format: "parquet"
    partition_by: ["extract_date"]
  quality:
    required: ["sku_id", "location_id", "on_hand_qty"]
    rules:
      - "on_hand_qty >= 0"
      - "extract_date <= today"
  load_mode: "INCREMENTAL"
  schedule: "0 6 * * *"   # 06:00 daily
framework/pipeline.py Python 
from abc import ABC, abstractmethod
import polars as pl

class Pipeline(ABC):
    """Base pipeline. Subclasses implement extract/transform."""

    def run(self) -> PipelineResult:
        df = self.extract()
        df = self.validate(df)
        df = self.transform(df)
        self.load(df)
        return PipelineResult(rows=df.height, status="ok")

    @abstractmethod
    def extract(self) -> pl.DataFrame: ...

    def validate(self, df: pl.DataFrame) -> pl.DataFrame:
        # Schema, null, range, and referential checks from config.
        return self.quality.apply(df)

    def load(self, df: pl.DataFrame) -> None:
        df.write_parquet(self.dest, partition_by=self.cfg.partition_by)

What I'd do differently

Start with the data contract

We shipped Bronze first and wrote contracts later. Codifying source schemas up front would have saved two weeks of downstream fixes.

Invest earlier in lineage

Our lineage was initially pipeline-local. An OpenLineage-compatible emitter from day one would have made cross-pipeline debugging much cheaper.

Delta, if we had the runtime

Plain Parquet was the right call for our runtime, but time-travel and ACID on Bronze would have removed a whole class of reprocessing risk.

Semantic layer as code

Business definitions lived in Power BI DAX. Moving them into a dbt-style semantic layer above Gold would have unlocked reuse outside BI.
