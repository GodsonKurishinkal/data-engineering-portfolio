01

The problem

Retail supply chains don't have one forecasting problem. They have 30,000 of them. A best-seller with steady daily demand is a different statistical beast than a spare part ordered once a month — and yet the legacy system applied the same simple moving average to all of them.

  • Frequent stockouts on fast movers — lost sales.
  • Excess inventory on slow movers — capital tied up in dead stock.
  • Planners overriding forecasts manually, hours per day.
  • No visibility into why a forecast missed.

The insight: pick the algorithm per SKU based on what its demand actually looks like, not what we wish it looked like.

ADI / CV² pattern classification

Two numbers describe any demand series well enough to pick a modelling approach:

  • ADI — Average Demand Interval. How often does demand occur?
  • CV² — Squared coefficient of variation. How variable is the size when it does?

Crossing the two thresholds (1.32 and 0.49) gives four quadrants — four patterns, each with its own winning algorithms.

Smooth
ADI < 1.32 · CV² < 0.49
Regular, predictable demand. Classic time-series territory.
ARIMA · Prophet · Holt-Winters
Erratic
ADI < 1.32 · CV² ≥ 0.49
Frequent but highly variable. Needs robust, non-parametric methods.
Random Forest · XGBoost
Intermittent
ADI ≥ 1.32 · CV² < 0.49
Infrequent but consistent size. Croston and friends.
Croston · SBA · TSB
Lumpy
ADI ≥ 1.32 · CV² ≥ 0.49
Rare and unpredictable. Plan safety stock; forecast humbly.
TSB · Bootstrap · Safety stock

05

Key decisions

Pattern first, model secondADR-001

Stacking everything into one deep model would have been a PR headline and an operational nightmare. Classifying first makes selection auditable — planners can see why Croston, not ARIMA, was chosen.

Trade-off: loses some accuracy from pattern misclassification. Caught by the fallback ensemble.
Explainability over accuracyADR-002

A 0.5% accuracy gain from a black-box model wouldn't have been worth the adoption penalty. Planners trust forecasts they can understand; surfacing pattern + model + drivers moved adoption far more than tuning ever could.

Trade-off: we leave a little accuracy on the table. Stockout reductions say it's fine.
Safety stock as a first-class outputADR-003

Every forecast ships with a prediction interval and a recommended safety stock. Point forecasts alone don't survive contact with real supply chain uncertainty.

Trade-off: more columns in the consumer table. Worth every byte.
Backtesting, alwaysADR-004

Rolling-origin backtests run on every release. A model can't win selection on training metrics alone — it has to beat the bench on the last N holdout windows.

Trade-off: slower release cadence. Caught a regression before it reached planners.

Implementation

The classifier is small, pure, and testable. Given a demand series, it emits a pattern label. Selection and forecasting are downstream of this one function.

forecast/pattern_classifier.py Python 
from enum import Enum
import pandas as pd

class DemandPattern(Enum):
    SMOOTH = "smooth"
    ERRATIC = "erratic"
    INTERMITTENT = "intermittent"
    LUMPY = "lumpy"

class PatternClassifier:
    """Classify a demand series by its ADI/CV² signature."""

    def __init__(self, adi_threshold: float = 1.32, cv2_threshold: float = 0.49):
        self.adi_threshold = adi_threshold
        self.cv2_threshold = cv2_threshold

    def classify(self, demand: pd.Series) -> DemandPattern:
        non_zero = (demand > 0).sum()
        if non_zero == 0:
            return DemandPattern.LUMPY

        adi = len(demand) / non_zero
        values = demand[demand > 0]
        cv2 = (values.std() / values.mean()) ** 2

        high_adi = adi >= self.adi_threshold
        high_cv2 = cv2 >= self.cv2_threshold

        if not high_adi and not high_cv2: return DemandPattern.SMOOTH
        if not high_adi and     high_cv2: return DemandPattern.ERRATIC
        if     high_adi and not high_cv2: return DemandPattern.INTERMITTENT
        return DemandPattern.LUMPY

What I'd do differently

Hierarchical forecasts

Some SKUs are too sparse to model on their own. Forecasting at category level and reconciling with MinT would have helped the long tail.

Champion / challenger

Model selection was per-release. A continuous challenger framework shadow-running the next-best model would have caught drift faster.

Intervals before points

Planners care more about "what's the worst case" than the point forecast. Leading with intervals from day one would have reframed the product.

Treat promotions as features

Promotional effects were layered as post-processing. Pushing them into the model as features earlier would have cleaned up the pipeline.
