01

The problem

Enterprise supply chains don't run on modern APIs. They run on 2000s-era ERPs, Java-applet warehouse systems, and reporting portals that only export to Excel. That data is critical — and it's locked behind screens.

  • Every morning, someone logged into four separate systems.
  • Each system needed a specific click-and-export dance to produce a report.
  • Exported files landed in Downloads, got dragged to SharePoint, got manually imported.
  • Total: 4+ hours of glorified copying before actual analysis could start.
  • Human errors silently broke reports downstream.

APIs were not coming. The systems couldn't be replaced. But a robot could do the clicking — and do it more reliably than a sleepy analyst at 06:30.

02

The system

A single abstract base class defines the shape of a bot: login, extract, persist, clean up. Each concrete bot fills in the system-specific parts. Retries, screenshots on failure, credential handling, and Teams alerts are inherited — not rewritten per bot.

04

Framework features

Retry with backoff
Exponential backoff around transient failures — network blips, slow renders, session timeouts.
Screenshot on failure
Every failure captures the browser DOM + a PNG. UI-drift debugging went from guesswork to pattern matching.
Credential vault
Encrypted at rest, rotated on schedule, never in logs. Per-bot role, least-privilege.
Smart waits
Element-visibility waits with per-step timeouts. Hard sleeps banned.
Alerts
Teams + email on failure or anomaly. Severity mapped to channel; pagers are for pages.
Audit trail
Full execution log — steps, timings, resulting records — written to a pipeline-log table for post-hoc review.
05

Key decisions

ABC pattern over copy-pasteADR-001

The first bot was a single script. By the fourth we'd have had four copies of retry logic, four ways of handling credentials, four debugging conventions. A shared abstract base class forced the right abstractions early.

Trade-off: writing a new bot requires knowing the framework. Mitigated with a template bot and docs.
Selenium for browsers, PyAutoGUI for the restADR-002

Selenium is precise when the target is a DOM. For Java applets and green-screen terminals, it gives up — PyAutoGUI + image matching fills the gap. Same bot shape, different engines.

Trade-off: PyAutoGUI bots are more fragile to UI redesigns. Caught quickly by the screenshot-on-failure hook.
Headless only when it helpsADR-003

Tempting to default everything to headless. Some legacy Java applets literally refuse to render without a visible window. The framework detects the target and switches mode; headless is a hint, not a contract.

Trade-off: a dedicated Windows runner instead of a container. Worth it for reliability.
Land raw, transform laterADR-004

Bots write exactly what they extract to Bronze, with metadata. All shaping and cleaning happens in the lakehouse. That way a single bot change doesn't ripple through every downstream model.

Trade-off: more columns to maintain at Bronze. The lineage benefit is worth it.
06

Implementation

The base class owns lifecycle and error handling. Subclasses provide the system-specific verbs. The pattern makes the shape of a bot uniform while letting the contents be as weird as the system requires.

rpa/base_bot.py Python 
from abc import ABC, abstractmethod
from selenium import webdriver
from selenium.webdriver.support.ui import WebDriverWait

class BaseBot(ABC):
    """Shared lifecycle for all RPA bots."""

    def __init__(self, config: BotConfig):
        self.config = config
        self.driver = None
        self.logger = setup_logger(self.__class__.__name__)

    def run(self) -> ExtractResult:
        """Main execution — initialise, login, extract, land, clean up."""
        try:
            self._initialize_driver()
            self._login()
            data = self._extract()
            self._save_to_bronze(data)
            return ExtractResult(success=True, records=len(data))
        except Exception as e:
            self._capture_screenshot()
            self._send_alert(str(e))
            raise
        finally:
            self._cleanup()

    @abstractmethod
    def _login(self) -> None: ...

    @abstractmethod
    def _extract(self) -> pd.DataFrame: ...

    def _wait_for(self, locator, timeout: int = 30):
        """Smart wait with element-visibility check."""
        return WebDriverWait(self.driver, timeout).until(
            EC.presence_of_element_located(locator)
        )

08

What I'd do differently

Playwright over Selenium

Playwright's auto-waiting and network interception would have simplified several Selenium workarounds. Selenium was the right choice then; not for a greenfield today.

Containerised runners

Running on a shared Windows host meant one bot's flakiness could starve another. Container-per-bot with pinned Chrome versions would have isolated failures.

Record, don't click

For stable flows, recording and replaying HAR traces would have been more reliable than UI automation. Reserve real clicking for actually dynamic UIs.

Push for real integrations

Bots are a pragmatic stopgap. We used the time they bought to push harder for proper APIs with the vendors — should have started that conversation on day one.
