01

The problem

In supply chain analytics, a bad row is not a reporting nuisance. A single corrupted inventory record can trigger an unnecessary purchase order. A missing dimension key can silently break a semantic model downstream. Data quality is an operations problem, not a BI problem.

  • Source systems routinely sent nulls where quantities should exist.
  • Dimension tables arrived with duplicate or missing keys, breaking joins.
  • Sales spikes from system glitches looked identical to real demand.
  • Gradual data drift went unnoticed until quarterly review.
  • When something broke, no one could say which records were at fault.

One detection method wasn't going to fix all of that. Schema checks miss behavioural issues. Business rules miss statistical drift. Z-scores miss referential integrity. We needed layers.

01
Schema validation
Gate: Bronze → Silver · Hard fail

Rule-based checks that enforce the data contract before a row enters Silver. If schema breaks, the pipeline refuses the batch — bad data never crosses the boundary.

  • Null checks on required fields
  • Type validation against declared schema
  • Referential integrity (FK → PK)
  • Range constraints (e.g. qty >= 0)
  • Primary-key uniqueness
02
Business rules
Gate: Silver → Gold · Flag + continue

Domain-specific validations that schema can't express. Rows failing a rule are flagged and routed to a quarantine partition; the batch still loads so operations aren't blocked by a handful of edge cases.

  • Cross-field consistency (ship_date ≥ order_date)
  • Domain logic (on-hand quantity can't be negative)
  • Reorder-point sanity (safety stock ≥ 0)
  • Conformed dimension matches across facts
  • Temporal logic (no future-dated transactions)
03
Statistical outliers
Continuous: Gold layer · Alert + log

Runs over Gold facts on a schedule. Catches individual outliers, trend breaks, and slow drift — the kind of issues that pass schema and rules but still tell you something is wrong.

  • Z-Score and Modified Z-Score (robust to outliers)
  • IQR fencing and MAD-based detection
  • Rolling coefficient of variation
  • Trend breaks and seasonal-deviation alerts
  • Volume spikes and week-over-week variance
03

Detection methods

Tier 3 runs multiple methods in parallel. Each method has a domain where it shines and a domain where it fails; picking one method per metric is a ticket to false positives.

Z-Score
Parametric · normal
Standard deviations from the mean. Fast, simple, works when data is roughly normal.
z = (x − μ) / σ
Modified Z-Score
Robust · outlier-resistant
Uses median and MAD. Doesn't get fooled by the outliers it's trying to find.
M = 0.6745(x − x̃) / MAD
IQR fencing
Non-parametric · quartile-based
Quartile bounds, no distribution assumption. Good default for skewed data.
[Q1 − 1.5·IQR, Q3 + 1.5·IQR]
Rolling CV
Time-aware · volatility
Coefficient of variation over a moving window. Flags regime changes.
CV = σ / μ (rolling)
Grubbs test
Statistical · single outlier
Formal hypothesis test for a single outlier in univariate data.
G = max|x − x̄| / s
DBSCAN
Density · multivariate
Density-based clustering for multivariate anomalies. Finds the "nothing nearby" points.
ε-neighbourhood density
04

Key decisions

Three tiers, not one smart modelADR-001

An ML-based anomaly detector would have been impressive and wrong. Schema errors don't need a neural net; they need a fast, boolean gate. Three focused layers are cheaper to reason about, debug, and tune.

Trade-off: each tier tuned separately. Easier than retuning one monolith.
Quarantine, don't deleteADR-002

Flagged rows go to a quarantine/ partition, not /dev/null. Sometimes the "anomaly" is a real event worth investigating — a demand spike, a new promotion, a store opening.

Trade-off: extra storage. Near-zero cost on Parquet.
Severity-aware alertingADR-003

Day one of Tier 3 produced 100+ Teams alerts. We tuned thresholds per-metric and tagged severity (INFO, WARN, PAGE), collapsing noise by 90%. If everyone's getting paged, no one is.

Trade-off: severity mapping is manual and per-metric. Worth the discipline.
Context over constantsADR-004

A 50% sales spike on Black Friday is the point. Time-aware thresholds — per weekday, per holiday, per season — dropped false positives by 60%.

Trade-off: needs a calendar of events. Maintained as a CSV checked into the repo.

Implementation

The Tier 3 detector is a small, configurable core. Each metric gets a config; the detector picks the right method and returns a boolean mask. No surprises.

quality/anomaly_detector.py Python 
from dataclasses import dataclass
from enum import Enum
import numpy as np

class Method(Enum):
    ZSCORE = "zscore"
    MODIFIED_ZSCORE = "modified_zscore"
    IQR = "iqr"

@dataclass
class Config:
    method: Method
    threshold: float = 3.0
    min_samples: int = 30

class AnomalyDetector:
    """Multi-method statistical anomaly detection."""

    def __init__(self, config: Config):
        self.config = config
        self._methods = {
            Method.ZSCORE: self._zscore,
            Method.MODIFIED_ZSCORE: self._modified_zscore,
            Method.IQR: self._iqr_fence,
        }

    def detect(self, data: np.ndarray) -> np.ndarray:
        """Return boolean mask of anomalies."""
        if len(data) < self.config.min_samples:
            return np.zeros(len(data), dtype=bool)
        return self._methods[self.config.method](data)

    def _modified_zscore(self, data: np.ndarray) -> np.ndarray:
        median = np.median(data)
        mad = np.median(np.abs(data - median))
        m = 0.6745 * (data - median) / (mad + 1e-10)
        return np.abs(m) > self.config.threshold

    def _iqr_fence(self, data: np.ndarray) -> np.ndarray:
        q1, q3 = np.percentile(data, [25, 75])
        iqr = q3 - q1
        return (data < q1 - 1.5*iqr) | (data > q3 + 1.5*iqr)

What I'd do differently

Declare rules as data

Tier-2 rules started as Python callables. Moving them to YAML earlier would have let analysts contribute without going through engineering.

Baselines from day one

We tuned thresholds reactively. A two-week baseline run with alerts muted would have given us better starting thresholds and fewer day-one pages.

SLOs, not thresholds

Framing detection as service level objectives ("99% of inventory rows pass schema") gave stakeholders clearer language than "z > 3."

Quarantine review loop

Quarantined rows accumulated. A weekly triage queue with labels ("real event" / "bad data" / "fix upstream") would have closed the feedback loop.
